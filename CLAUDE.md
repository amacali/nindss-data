# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this repo does

Scrapes daily notifiable-disease notification snapshots for Australia from the NINDSS PowerBI dashboard (https://nindss.health.gov.au/pbi-dashboard/) and archives them as JSON files in `data/`. A GitHub Actions workflow runs the scraper daily and commits the results — there is no application code to build or deploy, just a data pipeline.

## Commands

- Install dependencies: `npm install`
- Run the daily scraper (all-time totals): `node index.js` or `node index.js all-time` (writes `data/<report_date>_notifications.json`)
- Run the per-year breakdown (on request): `node index.js year` (current year only, default), `node index.js year 2019` (backfill/refresh a specific year), or `node index.js year all` (full history rebuild, ~2.4k requests) — writes `data/year/<report_date>_notifications.json`, assembled from the per-year files also in `data/year/` — see Architecture
- Run the monthly history (on request): `node index.js month` (current year+month only, default), `node index.js month 201907` (backfill/refresh one month), `node index.js month 2019` (backfill/refresh a whole year), or `node index.js month all` (full history rebuild — very expensive, ~30k requests) — writes `data/month/<report_date>_notifications.json`, assembled from the per-month files also in `data/month/` — see Architecture
- There are no tests, lint, or build steps configured (`npm test` is a stub that always fails).

## Architecture

The scraper is split across three files, all reverse-engineering the PowerBI embed API rather than using any public NINDSS API:

- `powerbi.js` — the shared DAX query client, with no CLI entry point of its own. Exports `STATE_CODES`/`MONTH_NAMES` plus:
  1. `getConfig()` — fetches the dashboard HTML page and extracts the `embedconfig` attribute from the `div.powerbi` element, base64-decoding it to get a report ID and embed token.
  2. `getToken()` — exchanges the embed token for a short-lived MWC token and capacity URI via PowerBI's `modelsAndExploration` endpoint.
  3. `getLatestUpdateDate()` — queries the `DataRefreshAEST` table (the same source backing the dashboard's "Last refreshed on" card) and returns both `reportDate` (`YYYYMMDD`, used for the filename/grouping key) and `lastRefreshed` (full AEST/AEDT timestamp, same underlying value with time preserved).
  4. `getCaseNumbers(..., mode)` — queries `NOTIFIABLE_EVENT_FACT` joined with `LOCATION_DIM`/`DISEASE_DIM`/`CASE_DIM` for per-state notification counts for one disease (restricted to Confirmed/Probable cases and excluding the `Hepatitis C (<24 months)` and `Unknown` disease groups). `mode` drives the query granularity AND return shape: `all-time` → `{ <state>: count }`; `year` → `{ <year>: { <state>: count } }` (the GROUPED, possibly-masked value — see point 5 for why `index.js`'s own `year` build doesn't use this); `month` → `{ <year>: { <month>: { <state>: count } } }` (also GROUPED/masked — see point 6). **Each mode is queried at its own granularity, never derived from a finer one, on purpose:** the dashboard masks any cell `<5`, and summing masked finer cells accumulates the loss (COVID-19's lifetime total: 12,302,011 queried as `all-time`, 12,302,009 summed from `year`, 12,301,939 summed from `month`). Used by `index.js`'s `all-time` mode and `legacy.js`'s `year`-mode query — `index.js`'s own `year`/`month` builds use the cumulative helpers below instead.
  5. `getYearCumulativeTotal(capacityUri, token, diseaseName, maxYear)` — exported helper, used only by `index.js`'s `buildYearOutput` (NOT by `getCaseNumbers` or `legacy.js`). Same STATE-on-primary-axis shape as `all-time`, plus one extra `DAX_Year <= maxYear` filter (`ComparisonKind: 4`), so it returns a **running total through `maxYear`**, not that year's own count. A running total is far less likely to fall under the `<5` mask than any single year's grouped count (confirmed empirically: Measles' grouped `year`-mode query masked ACT/SA/TAS in 2019 and VIC/WA in 2020 to 0, while diffing consecutive cumulative totals recovered the true nonzero counts). PowerBI's DAX filter engine accepts arbitrary `In`/`Comparison` filters on `DAX_Year` the same way it already does for `STATE`/`CONFIRMATION_STATUS` — confirmed by live testing against the dashboard.
  6. `getMonthCumulativeTotal(capacityUri, token, diseaseName, year, month)` — exported helper, used only by `index.js`'s `buildMonthOutput`. Returns a running total through (`year`,`month`), analogous to `getYearCumulativeTotal` but a genuinely harder problem: a within-year-only window (e.g. "all months of just this year so far") is exactly as maskable as the plain grouped query, because PowerBI masks based on the resulting aggregate VALUE, not the query shape — confirmed empirically (a masked single-year total stayed masked whether expressed as a year group-by or as an equivalent "all 12 months of this year" month-name filter). What actually resists masking is a running total spanning the ENTIRE history up to the target month, expressed as one query with a genuine `Or` across two sub-filters: `(DAX_Year <= year - 1) OR (DAX_Year = year AND Month IN [Jan..month])`. Summing two SEPARATELY-queried (and separately-maskable) pieces in JS does NOT work — each piece can already be masked before the sum happens, silently losing the same information a plain query would. PowerBI's DAX query JSON supports arbitrary `Or`/`And` combinators the same way it supports `Not`/`In`/`Comparison` — confirmed by live testing: results matched `getYearCumulativeTotal` exactly at year boundaries (December of each year), and revealed genuine new unmasked data mid-year (a Measles ACT case in H1 2019 that every single-year query had masked to 0).
- `legacy.js` — **deprecated, slated for removal, and intentionally frozen: it serves an old site and its output format must not change.** `writeLegacyCases(capacityUri, token, reportDate, diseaseNames)` re-queries each disease at `year` granularity via `powerbi.js`'s `getCaseNumbers` (the plain grouped query, unaffected by anything in `index.js`) and writes `data/<reportDate>_cases.json`, a flat array of `{ REPORT_DATE, DISEASE, YEAR, CODE, CASES }` records — the pre-rewrite schema, kept only for old consumers. This is the entire legacy surface: deleting `legacy.js` and its one call site in `index.js` (`getDiseaseList`'s `all-time` branch) removes it cleanly. Only produced on `all-time`-mode runs (the daily cron); `year`/`month` on-request runs don't touch it. Do not refactor this file's output shape or point it at the cumulative helpers — its whole purpose is to stay byte-for-byte compatible with what the old site already expects.
- `index.js` — the entry point. First CLI arg selects the mode (defaulting to `'all-time'`); `year`/`month` take an optional second `scopeArg` (see below). `getDiseaseList(mode, scopeArg)` queries the `DISEASE_DIM` table for all disease names, then dispatches: `all-time` falls through to a plain loop calling `getCaseNumbers(..., 'all-time')` per disease and writes `data/<reportDate>_notifications.json` (plus `legacy.js`'s deprecated output); `year`/`month` each delegate entirely to their own build function, writing exclusively under `data/year/`/`data/month/` — neither writes anything at the top level (`data/<reportDate>_notifications_year.json`/`_month.json` no longer exist).
  - `buildYearOutput(capacityUri, token, diseases, reportDate, lastRefreshed, yearsToFetch)` — writes one file per `DAX_Year` in `yearsToFetch` under `data/year/<year>_notifications.json` (cumulative per-state totals via `getYearCumulativeTotal` across every disease), then rebuilds `data/year/<reportDate>_notifications.json` (the combined snapshot, named by the 8-digit scrape date rather than a 4-digit year, living in the SAME directory as the per-year caches) by reading and concatenating **every** per-year cache file present on disk — not just the year(s) fetched this run — so the combined snapshot always reflects the fullest known history at no extra request cost. Because both file kinds share one directory and both end in `_notifications.json`, the cache scan matches strictly on the 4-digit-year form (`/^\d{4}_notifications\.json$/`) so it never picks up the 8-digit combined file as if it were a year. `yearsToFetch` is computed in `getDiseaseList` from `scopeArg`: no arg → `[currentYear]` (default, cheap, ~1 request/disease); a specific year (e.g. `2019`) → `[2019]` (targeted backfill/refresh); `'all'` → every year from `YEAR_FLOOR = 1990` (matching the floor already baked into every `powerbi.js` query) through the current year (full history rebuild, ~2.4k requests). **No year is ever skipped/reused from an existing cache file** — whatever year(s) are targeted are always fetched live and their cache file overwritten; only years NOT targeted this run are left as-is (read back in for the combined-file assembly, not re-queried). Both the per-year cache files and the combined file store **cumulative totals through that year, not per-year deltas** — deltas are computed downstream in the database.
  - `parseMonthScope(scopeArg, currentYear, currentMonth)` — turns `scopeArg` into a list of `{ year, month }` periods for `buildMonthOutput`: no arg → `[{currentYear, currentMonth}]`; `'YYYYMM'` (6 digits) → that single period; `'YYYY'` (4 digits) → every month of that year (Jan through `currentMonth` if it's the current year, else Jan–Dec); `'all'` → every month from `YEAR_FLOOR` through the current month (full history — ~37 years × 12 months × ~67 diseases ≈ 30k requests, deliberately large enough that it should never be run casually).
  - `buildMonthOutput(capacityUri, token, diseases, reportDate, lastRefreshed, periodsToFetch)` — the `month`-mode mirror of `buildYearOutput`: one file per `{year, month}` in `periodsToFetch` under `data/month/<YYYYMM>_notifications.json` (cumulative per-state totals via `getMonthCumulativeTotal`), then a combined `data/month/<reportDate>_notifications.json` rebuilt from every 6-digit-named cache file on disk (same collision-avoidance trick as `buildYearOutput`, just on a 6-digit vs 8-digit filename split instead of 4-digit vs 8-digit).

All PowerBI requests are raw `fetch` calls with hand-built DAX query JSON bodies (`SemanticQueryDataShapeCommand`) sent as strings — there is no query builder abstraction. If PowerBI changes its dataset/report IDs or query shape, these request bodies (`DatasetId`, `ReportId`, `VisualId`, column/entity names) are what break and need updating.

The three modes also produce **two different response LAYOUTS**, because PowerBI rejects a secondary axis with no primary (`SecondaryGroupsWithoutPrimary`): `year`/`month` keep STATE on the *secondary* axis (the per-row `X` array, with the period(s) as primary rows), but `all-time` has no period dimension, so STATE moves to the *primary* axis — each `DM0` row is one state, projected as `C: [state, measure]`, and there is no `X` array or `SH` state list at all.

Response parsing relies on PowerBI's compact `dsr.DS[0]` result-set format (`PH`/`DM0`/`SH`/`DM1`). Two distinct sparse-encoding schemes are in play and are easy to conflate:
- **Measure sparsity** (the `X` array per row, one entry per state, in `year`/`month`): a state's `M0` is omitted when it repeats the previous state's value — see the "check if value exists, otherwise repeat" logic in `getCaseNumbers`.
- **Row sparsity** (the `DM0` rows themselves, whenever more than one value is projected onto a row): a row's `R` field is a bitmask marking which projections repeat from the previous row — only the *changed* ones are consumed off that row's `C` array, in order; the rest carry forward. This drives three cases: `month` projects dictionary-encoded `[year, month]` (`ds0.ValueDicts.D0`/`D1`), decoded to track `[year, month]`; `all-time` projects `[state, measure]` (the measure repeats over runs of equal counts, e.g. long stretches of 0); and `year` projects a single primary dimension, so there is no dictionary/bitmask at all — the year is read straight off the row as `G0`. Projecting an additional hierarchy level shifts PowerBI's `G`-numbering for every dimension after it — this is why the STATE secondary-axis key is `G1` in `year` mode but `G2` in `month` mode (adding Month bumps it), the most common source of silent breakage when adjusting these queries.

## Data output

All `_notifications*` files use a flat, MySQL-friendly shape chosen so they load via a single `JSON_TABLE('$.rows[*]' …)` call. State counts are inlined in the fixed `STATE_CODES` order defined in `powerbi.js`; AUS/national is excluded by the query. Each file is queried at its own granularity and is **not** an exact sum of a finer one (coarser = less `<5` masking, so slightly higher and more accurate — see the note under `getCaseNumbers`).

- `data/YYYYMMDD_notifications.json` (daily, all-time totals) — `{ report_date, last_refreshed, columns: ["disease",<8 states>], rows: [ [disease, ...8 counts], … ] }`. 67 rows (one per disease), no `year` column.
- `data/YYYYMMDD_cases.json` (daily, written alongside `_notifications.json` on `all-time`-mode runs) — **deprecated, see `legacy.js`.** A flat array of `{ REPORT_DATE, DISEASE, YEAR, CODE, CASES }` records (year granularity, one per disease+year+state), the pre-rewrite schema kept for old consumers.
- `data/year/` — everything `year` mode produces; NOT under the top-level `data/<reportDate>_...` naming `all-time` uses.
  - `data/year/<year>_notifications.json` — one per `DAX_Year` (bare 4-digit year, e.g. `2019_notifications.json`), `{ year, columns: ["disease",<8 states>], rows }`. Cumulative per-state totals through that year (`DAX_Year <= year`), not per-year deltas — see `getYearCumulativeTotal`/`buildYearOutput` above. Only rewritten when explicitly targeted by `node index.js year [Y|all]`; otherwise left as-is.
  - `data/year/<reportDate>_notifications.json` — the combined snapshot (8-digit scrape date, e.g. `20260718_notifications.json`), `columns: ["disease","year",<8 states>]`, up to ~2.4k rows (67 diseases × ~36 years) once every year has been backfilled at least once. Rebuilt every `year`-mode run from every `<year>_notifications.json` present in the same directory, regardless of which year(s) that run actually fetched live. Downstream consumers compute the per-year delta themselves.
- `data/month/` — everything `month` mode produces, mirroring `data/year/` exactly one level finer.
  - `data/month/<YYYYMM>_notifications.json` — one per (year, month) (bare 6-digit form, e.g. `201907_notifications.json`), `{ year, month, columns: ["disease",<8 states>], rows }`. Cumulative per-state totals through that month (spanning full history, not just that year — see `getMonthCumulativeTotal` above), not per-month deltas. Only rewritten when explicitly targeted by `node index.js month [YM|Y|all]`; otherwise left as-is.
  - `data/month/<reportDate>_notifications.json` — the combined snapshot (8-digit scrape date), `columns: ["disease","year","month",<8 states>]`, up to ~28k rows once every month has been backfilled. Rebuilt every `month`-mode run from every `<YYYYMM>_notifications.json` present in the same directory, regardless of which period(s) that run actually fetched live. The finest granularity, so it took the most work to make trustworthy — see `getMonthCumulativeTotal`'s header comment for why a naive within-year sum doesn't avoid masking the way `year` mode's cumulative approach does.

New files accumulate via the CI workflow; existing files are never rewritten by hand — except everything under `data/year/` and `data/month/`, which are long-lived and overwritten in place: a per-period cache file (`<year>_notifications.json` / `<YYYYMM>_notifications.json`) only when that period is explicitly targeted by `node index.js year [Y|all]` / `node index.js month [YM|Y|all]` (every other period's file is left untouched, just read back in to assemble the combined output), and the combined `<reportDate>_notifications.json` in each directory on every corresponding-mode run. Backfilling a full range (`node index.js year all` / `node index.js month all`) is a manual, deliberate, expensive operation — it is NOT run automatically by the CI cron, which only ever passes `year`/`month` with no scope arg (current period only). `month all` in particular is ~30k requests and should essentially never be run without a specific reason. No historical data predates the current schema — old snapshots were deleted rather than migrated. (The bare `_notifications.json` name previously held year totals, and before that an even earlier same-day iteration nested `data.<disease>.<year>.<month>.<code>`, dropped because nested object keys can't be unnested by MySQL's `JSON_TABLE`.)

## CI

`.github/workflows/main.yml` runs on a twice-daily cron and via manual dispatch: checkout, `npm install`, run the scraper, then commits and pushes any new/changed files in `data/` directly to `main`. The scheduled cron passes no argument (all-time totals); the manual dispatch exposes a `mode` choice input (`all-time`/`year`/`month`) forwarded to `node index.js` to (re)generate the corresponding file — always with no scope arg, so `year`/`month` dispatches only ever refresh the current period, never a full backfill.
